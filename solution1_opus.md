# Обновлённая структура выполненного тестового задания: Network Effects в финтехе

---

## Ключевая рамка через весь документ

> Network effect отвечает на вопрос: растёт ли ценность продукта для пользователя из-за активности, наличия или вклада других пользователей?

Каждый раздел пропускается через этот фильтр. Если ценность не растёт от других пользователей — это не сетевой эффект, а что-то другое (масштаб, маркетинг, кросс-продажи, lock-in).

Два типа эффектов, которые мы различаем всюду:

- Прямой: больше пользователей → больше ценность для каждого (WhatsApp: «там мои родители, меня бесит, но я пользуюсь»)
- Непрямой: одни пользователи создают ценность для других (UGC-контент; «раздели счёт» — один юзер создаёт контекст, другие приходят за контекстом)

Два измерения пользователей: новые / старые — и нужно понимать, кто получает ценность от кого.

## ЧАСТЬ I. ПОДГОТОВКА

### 1. Концептуальная рамка: что делает продукт сетевым

Формат: 1–2 страницы + схема

Содержание:

- Определение через центральный вопрос: «Растёт ли ценность для юзера X из-за того, что юзер Y существует / активен / что-то вносит?»
- Чёткое разделение: прямой vs непрямой эффект
  - Прямой: ценность = f(количество пользователей). Телефон, мессенджер, платёжная сеть. Механика: я не могу получить ту же ценность без присутствия конкретных людей
  - Непрямой: ценность = f(вклад одних пользователей для других). UGC, маркетплейсы, «раздели счёт». Механика: один юзер создаёт артефакт / контекст — другой юзер приходит за этим артефактом
- Матрица: кто получает ценность?

|  | Новый пользователь получает | Старый пользователь получает |
|--|---|---|
| От новых | Расширение сети (прямой) | Рост полезности существующей сети (прямой) |
| От старых | Готовый контент / контекст / данные (непрямой) | Углубление связей / данных (непрямой) |

- Что НЕ является сетевым эффектом (частые путаницы):
  - Масштаб (больше клиентов → дешевле юнит, но ценность для юзера не растёт)
  - Кросс-продажи (экосистема одного юзера)
  - Вирусный маркетинг (механика распространения, но не механика ценности)
  - Lock-in / switching costs (юзер остаётся не потому что ценно, а потому что дорого уйти)

Визуал: Схема — дерево решений: «Это сетевой эффект?» → «Прямой или непрямой?» → «Кто кому создаёт ценность?»

### 2. Разбор успешных кейсов через призму прямой/непрямой ценности

Формат: таблица с глубоким разбором + аналитика

| Продукт | Core interaction | Тип NE | Кто создаёт ценность для кого | Почему старый юзер не уходит | Почему новый юзер приходит |
|---------|-----------------|--------|-------------------------------|-------|--------|
| WhatsApp | Сообщение → конкретному человеку | Прямой | Каждый юзер — ценность для своих контактов | «Там мои родители» — граф связей невоспроизводим | Все мои контакты уже там |
| Uber | Запрос → матчинг с водителем | Непрямой (кросс-сайд) | Водители создают ценность для пассажиров и наоборот | Привычка + плотность водителей = минимальное ожидание | Быстрая подача (= много водителей) |
| Instagram | Пост → лайк/follow | Непрямой (UGC) | Авторы создают контент для потребителей; потребители создают аудиторию для авторов | Мой контент, мои подписчики, моя аудитория | Контент уже есть + мои друзья уже здесь |
| Visa/MC | Платёж картой | Непрямой (кросс-сайд) | Кардхолдеры → мерчанты; мерчанты → кардхолдеры | Все мерчанты принимают | Все покупатели имеют карту |
| Splitwise | Добавить расход в группу | Прямой + непрямой | Один юзер создаёт контекст долга → другие приходят закрывать / видеть | История долгов и расчётов — невоспроизводима | Друг прислал ссылку «ты мне должен» |
| M-Pesa | Перевод по номеру телефона | Прямой | Каждый участник сети = потенциальный получатель/отправитель | Вся торговля идёт через M-Pesa | Невозможно участвовать в экономике без M-Pesa |
| WeChat Pay | Оплата → красные конверты → мини-аппы | Прямой + экосистемный | Социальный граф → финансовые действия; мини-аппы создают ценность для юзеров | Вся жизнь внутри: чаты + деньги + сервисы | Не быть в WeChat = не быть в обществе |

**Аналитика:**

- **Паттерн:** все успешные кейсы имеют core interaction, для которого НЕОБХОДИМ другой человек
- **Прямой NE** побеждает в коммуникациях и P2P-платежах
- **Непрямой NE** побеждает в маркетплейсах и UGC
- **Самые мощные:** комбинация обоих (WeChat, Instagram)
- **Для финтеха критично:** кто создаёт контекст первым? В Splitwise один человек «создаёт долг» — остальные приходят за контекстом. Это непрямой NE через финансовое действие.

### 3. Предопределён ли успех

**Формат:** аналитическое эссе с контрфактуалами

**Структура:**

**3.1. Провалы с «правильной» архитектурой**

- Google+ (идеальные ресурсы, нет уникального core interaction)
- Clubhouse (идеальный момент, не масштабируемый непрямой NE — создавали контент только элиты)
- Facebook Pay / Libra (огромный граф, но нет финансового core interaction — деньги не стали «социальными»)

**3.2. Успехи с «неправильными» стартовыми условиями**

- M-Pesa (нет смартфонов, нет банковской инфраструктуры → стал инфраструктурой)
- Venmo (странная идея — лента платежей как соцсеть → зацепила студентов)

**3.3. Ключевые выводы**

- Архитектура создаёт потенциал сетевого эффекта
- Момент + контекст определяют, реализуется ли потенциал
- Критическая масса в минимальном юните (не «миллион юзеров», а «5 друзей в одной группе») — это то, что запускает маховик
- **Для нашего кейса:** не нужна массовая критическая масса. Нужна плотность в микрогруппе. Это радикально снижает барьер запуска.

### 4. Маркетинг vs сетевые эффекты

**Формат:** сравнительный анализ через центральный вопрос

**Ключевой тест:** если убрать маркетинговый бюджет — продолжает ли продукт расти?

- Да → сетевой эффект
- Нет → купленный рост

| Критерий | Купленный рост (маркетинг) | Сетевой эффект |
|----------|---------------------------|----------------|
| Источник новых юзеров | Рекламный бюджет | Действия существующих юзеров |
| Что происходит при отключении бюджета | Рост останавливается | Рост продолжается или замедляется, но не прекращается |
| CAC со временем | Стабилен или растёт (аукционная инфляция) | Снижается (каждый юзер приводит следующего дешевле) |
| Retention | Зависит от продукта | Зависит от плотности сети вокруг юзера |
| Защитный ров (moat) | Нет — конкурент перекупит внимание | Да — юзер «залочен» связями и контекстом |
| Ценность нового юзера для старых | Никакой | Прямая (прямой NE) или опосредованная (непрямой NE) |

**Разбор гибридов:**

- **PayPal:** $10 за реферала = маркетинг. Но привязка к eBay = непрямой NE (больше покупателей с PayPal → мерчанты подключают → больше покупателей). Маркетинг дал критическую массу, NE удержал.
- **Revolut:** агрессивный маркетинг + рефералка. Но NE почти нет — ценность для юзера одинакова, есть ли у его друзей Revolut или нет. Прямой тест: «стал ли мой Revolut полезнее оттого, что коллега тоже завёл Revolut?» — скорее нет.
- **Splitwise:** минимальный маркетинг. Каждое разделение счёта = встроенное приглашение. Прямой тест: «стал ли мой Splitwise полезнее оттого, что подруга тоже его использует?» — абсолютно да.

**Вывод для кейса:** нам нужна архитектура, где core action содержит в себе приглашение. Не «пригласи друга и получи бонус», а «чтобы сделать Х, тебе нужен другой человек, и самый простой способ его подключить — через наш продукт».

### 5. Сетевые эффекты в финансах сегодня — честная карта

**Формат:** карта с жёсткой фильтрацией через определение NE

**Фильтр для каждого примера:** «Растёт ли ценность для юзера А из-за активности / наличия / вклада юзера Б?»

**✅ Настоящие сетевые эффекты в финансах:**

| Что | Тип NE | Как работает | Прямой тест |
|-----|--------|-------------|-------------|
| Visa/MC (карточная сеть) | Непрямой | Больше мерчантов → полезнее карта; больше карт → мерчанты подключаются | Карта бесполезна без мерчантов, мерчант не подключит терминал без карт |
| M-Pesa | Прямой | P2P-переводы: полезность = f(кол-во людей в сети) | Бессмысленно иметь M-Pesa, если получатель не имеет |
| Venmo (лента платежей) | Прямой + непрямой (UGC) | Вижу, что друзья платят → социальное давление + удобство P2P | «Venmo me» стало глаголом → работает только если получатель тоже на Venmo |
| WeChat Pay (красные конверты) | Прямой | Социальный ритуал → нужен контрагент в сети | Не пошлёшь конверт человеку вне WeChat |
| Splitwise | Прямой + непрямой | Один создаёт расход → остальные видят долг → все считают через систему | Splitwise бессмыслен одному |
| DeFi/ликвидность | Непрямой | Больше провайдеров ликвидности → лучше цены для трейдеров → больше трейдеров → больше комиссий → больше провайдеров | Пул без ликвидности бесполезен |

❌ Что кажется NE, но не является (тест проваливают):

| Что | Почему не NE | Что это на самом деле |
|-----|-------------|----------------------|
| Банковская экосистема (Тинькофф, Сбер) | «Стал ли мой Тинькофф полезнее, потому что сосед тоже клиент?» — Нет | Кросс-продажи одному юзеру + экономия от масштаба |
| Кэшбэк-программы | Ценность для меня не зависит от других юзеров | Стимулирование покупки (маркетинг) |
| Кредитный скоринг на данных | Банк получает ценность от данных юзеров, но юзер не получает ценность от других юзеров | Data asset банка, не NE для пользователя |
| Реферальные программы банков | «Пригласи друга — получи 500 рублей» | Покупка канала привлечения (маркетинг) |
| Биржевой ордербук | Больше участников → лучше ликвидность, но воспринимается как инфраструктура | Граница: можно считать NE, но юзер не осознаёт связь |

Вывод — пустое поле: социальный слой финансовых взаимодействий между знакомыми людьми (не анонимная инфраструктура типа Visa и не один-на-один перевод через банк) практически пуст. Существующие решения:

- Splitwise — только учёт долгов, не движение денег
- Venmo — движение денег + социальная лента, но примитивное взаимодействие
- Банковские переводы — peer → bank → peer (n-1-n), нет понятия «группы», «контекста», «совместного действия»

Окно возможности: создать продукт, где финансовое взаимодействие между людьми — это core interaction, а не побочная функция банковского приложения.

### 6. Плейбук — повторяемые паттерны

Формат: структурированный плейбук из 8 шагов, каждый проверяется через определение NE

Шаг 1. Привязаться к существующему социальному графу

- Не строить сеть с нуля — импортировать
- Контакты телефона, группы в мессенджерах, совместные транзакции в банке
- Проверка через NE: граф = потенциальная сеть, в которой может возникнуть ценность от других

Шаг 2. Определить минимальный сетевой юнит

- Наименьшая группа, в которой уже возникает ценность
- Пара (2), друзья за ужином (3–5), семья (3–7), соседи по квартире (2–4)
- Проверка: «Ценен ли продукт для этих 3 человек, даже если во всём мире больше никто им не пользуется?» — если да, юнит найден

Шаг 3. Спроектировать core interaction, для которого НУЖЕН другой человек

- Не «я перевожу деньги» (могу через любой банк), а «мы вместе делаем X»
- X = делим расходы / копим на общую цель / видим общий бюджет / голосуем кошельком
- Проверка через NE: «Можно ли совершить core interaction в одиночку?» — если да, NE нет

Шаг 4. Встроить приглашение в core action (не рядом, а внутри)

- Чтобы совершить действие, нужно подключить другого человека
- «Раздели счёт» → нужно добавить участников → приглашение
- «Создай общую цель» → нужно пригласить партнёра → приглашение
- Проверка: приглашение — это часть UX основного действия или отдельная кнопка? Если отдельная — это маркетинг, не NE

Шаг 5. Сделать так, чтобы новый юзер получал ценность мгновенно

- Новый юзер приходит не в пустоту, а в контекст, созданный старым юзером (непрямой NE)
- «Тебе пришёл запрос на 500 рублей за ужин» — контекст уже есть, ценность мгновенная
- Проверка: «Через сколько секунд новый юзер понимает, зачем он здесь?» — если больше 10 секунд, плохо

Шаг 6. Создать накапливаемый артефакт

- Данные, которые растут со временем и которые невозможно воспроизвести
- История совместных расходов, паттерны трат группы, прогресс общей цели
- Проверка: «Что теряет юзер, если уходит после 6 месяцев использования?» — если ничего, NE слабый

Шаг 7. Обеспечить мультигрупповое участие

- Один юзер = участник нескольких групп (семья + друзья + коллеги)
- Каждая новая группа = новый кластер, куда юзер приносит продукт
- Проверка: «Может ли юзер быть мостом между двумя кластерами?» — если да, кросс-кластерное распространение

Шаг 8. Различать эффекты для новых и старых юзеров

| | Что получает новый юзер | Что получает старый юзер |
|--|---|---|
| От прямого NE | Доступ к существующей сети контактов | Ещё один контакт в сети → ещё удобнее |
| От непрямого NE | Готовый контекст (долг, цель, бюджет) | Больше участников → богаче данные → точнее картина |

### 7. Нереализованный потенциал

Формат: 5 идей, каждая с анализом типа NE, барьеров и проверкой через определение

Идея 1: Финансовый мультиплеер — совместный бюджет в реальном времени

- Описание: как Google Docs для бюджета. Два или более человека видят общие расходы, доходы, лимиты в реальном времени. Не совместный счёт (это юридическая конструкция), а совместное представление (view) поверх индивидуальных счетов.
- Тип NE: прямой (ценность = f(количество участников, видящих общую картину)) + непрямой (каждая транзакция одного юзера = информация для другого)
- Проверка: «Стал ли мой продукт полезнее, потому что партнёр тоже подключился?» — однозначно да: я вижу полную картину только

когда подключены оба

- Кто → кому: старый юзер → новому (готовая структура бюджета); новый юзер → старому (полнота данных)
- Барьер: банки мыслят аккаунтами, а не связями. Архитектура core banking не поддерживает «shared view поверх разных счетов». Ментальный барьер: «мои деньги — моё дело»
- Что мешает: технологический (API, интеграция нескольких банков) + культурный (прозрачность финансов)

Идея 2: Социальный контекст платежей

- Описание: каждый платёж можно обогатить контекстом — для кого, зачем, в рамках чего. Не просто «перевод 1500 руб Маше», а «1500 руб → Маша → наш ужин 15.03 → я заплатил за всех → Маша, Петя и Коля должны по 500». Платёж становится социальным объектом.
- Тип NE: непрямой — один юзер создаёт контекст (добавляет описание, привязывает к событию), другие юзеры приходят за контекстом (видят, сколько должны, за что, кому)
- Проверка: «Растёт ли ценность от того, что другие тоже используют?» — да: чем больше участников размечают свои платежи, тем полнее картина для всех
- Барьер: неочевидная ценность для первого юзера, который должен размечать платёж. Нужна мгновенная отдача.

Идея 3: Групповая покупательная способность (Collective Bargaining)

- Описание: группа людей (соседи, коллеги, друзья) объединяет покупательную способность для получения лучших условий: групповые скидки на подписки, оптовые цены, коллективные ставки по кредитам / страховкам
- Тип NE: прямой — чем больше людей в группе, тем лучше условия для каждого. Буквально: ценность = f(размер группы)
- Проверка: «Один юзер получает скидку 5%. Десять юзеров — 15%. Сто юзеров — 30%.» — классический прямой NE
- Кто → кому: каждый новый юзер → всем существующим (улучшает условия для всех)
- Барьер: юридическое оформление, координация, доверие. Партнёрства с поставщиками. Сложная операционка.

Идея 4: Peer-to-peer финансовое менторство

- Описание: юзеры, которые хорошо управляют бюджетом, делятся шаблонами, правилами, паттернами с другими. Не финансовый советник, а «мой друг, который всегда всё считает, и я хочу как он»
- Тип NE: непрямой (UGC) — одни юзеры создают ценный контент (шаблоны бюджетов, правила, лайфхаки), другие потребляют
- Проверка: «Стал ли продукт полезнее, потому что другой юзер создал шаблон бюджета?» — да
- Барьер: стигма вокруг обсуждения личных финансов. Мотивация создавать контент.

Идея 5: Финансовые ритуалы (Social Money Rituals)

- Описание: привязка денежных действий к социальным ритуалам. Красные конверты WeChat — это ритуал. А что, если: «Конверт на день рождения с подписями от группы друзей», «Еженедельный family pool — каждое воскресенье семья скидывается на общее», «Челлендж-копилка — группа друзей соревнуется, кто больше отложит за месяц»
- Тип NE: прямой (ритуал имеет смысл только в группе) + непрямой (один юзер создаёт ритуал → другие подхватывают формат)
- Проверка: «Можно ли совершить этот ритуал в одиночку?» — нет. По определению групповое действие.
- Барьер: нужно создать привычку. Ритуал работает только при регулярности. Высокий churn в первые недели.

### 9. Теория и классификация — таксономия NE в финансах

**Формат:** классификатор + схема

**Классификатор сетевых эффектов для финансовых продуктов:**

**По типу ценности:**

| Тип | Механика | Пример в финансах | Новый → старый / старый → новый |
|-----|---------|-------------------|-------------------------------|
| **Прямой коммуникационный** | Больше людей = больше возможных контрагентов | P2P-переводы внутри сети | Новый юзер = новый контрагент для старых |
| **Прямой групповой** | Ценность возникает в замкнутой группе, растёт с каждым участником | Совместный бюджет семьи, split-расходы друзей | Новый участник = полнее картина для всех; старые участники = готовый контекст для нового |
| **Прямой коллективный** | Чем больше участников, тем лучше условия для каждого | Групповая покупательная способность, коллективное страхование | Каждый новый = улучшение условий для всех |
| **Непрямой контекстный** | Один юзер создаёт контекст (расход, цель, событие) → другие приходят за контекстом | «Раздели счёт» — инициатор создаёт контекст долга → участники приходят закрывать | Старый → новому (контекст); новый → старому (закрытие долга) |
| **Непрямой UGC** | Одни юзеры создают контент/артефакты → другие потребляют | Шаблоны бюджетов, финансовые лайфхаки, паттерны трат | Старые авторы → новым потребителям; новые потребители → авторам (аудитория, фидбэк) |
| **Непрямой кросс-сайд** | Две разные стороны создают ценность друг для друга | Мерчанты ↔ кардхолдеры; заёмщики ↔ кредиторы в P2P-lending | Каждая сторона делает другую сторону ценнее |
| **Экосистемный / платформенный** | Каждый новый сервис / интеграция увеличивает ценность всей системы | Финансовый суперапп с мини-сервисами | Новые сервисы → всем юзерам; больше юзеров → привлекает новые сервисы |

**По динамике новый / старый:**

             Новый юзер получает ценность от:
             ┌──────────────────┬──────────────────┐
             │   Старых юзеров  │   Других новых   │
             │  (непрямой NE)   │   (прямой NE)    │
             │                  │                  │
             │ Готовый контекст │ Расширение сети  │
             │ Готовый контент  │ Больше контрагент│
             │ Готовая структура│ Лучшие условия   │
             └──────────────────┴──────────────────┘

             Старый юзер получает ценность от:
             ┌──────────────────┬──────────────────┐
             │   Новых юзеров   │   Других старых  │
             │  (прямой NE)     │  (непрямой NE)   │
             │                  │                  │
             │ Ещё один контакт │ Углубление данных│
             │ Лучшие условия   │ Богаче контент   │
             │ Полнее картина   │ Точнее паттерны  │
             └──────────────────┴──────────────────┘

**По силе / защитному рву:**

| Уровень | Характеристика | Пример | Сложность копирования |
|---------|---------------|--------|----------------------|
| **1. Слабый** | NE есть, но юзер легко переключается | Split-калькулятор (Splitwise без истории) | Низкая — конкурент делает то же за месяц |
| **2. Средний** | NE + накопленные данные | Совместный бюджет с историей за год | Средняя — данные не переносимы, но можно начать заново |
| **3. Сильный** | NE + данные + социальный граф внутри продукта | Финансовые группы с историей, ритуалами, привычками | Высокая — нужно перетащить всю группу целиком |
| **4. Критический** | NE + данные + граф + инфраструктурная зависимость | M-Pesa, WeChat Pay — вся экономика завязана | Практически невозможно — требуется замена инфраструктуры |

**10.2. Связи, которые создают NE**

| Связь | Тип NE | Почему создаёт ценность |
|-------|--------|------------------------|
| User ↔ User (внутри Group) | Прямой групповой | Каждый новый участник = полнее картина для всех |
| User → Action → Context | Непрямой контекстный | Один юзер создаёт контекст → другие приходят за ним |
| User → Shared View | Прямой групповой | Shared view бесполезен для одного |
| Action → Group (история) | Непрямой (данные) | Чем больше действий → тем ценнее история → тем сложнее уйти |
| User (Group A) ↔ User (Group B) | Кросс-кластерный | Один юзер = мост между группами → распространение |
| User → Reputation | Непрямой | Репутация имеет смысл только в контексте других людей |

**10.3. Ключевые архитектурные принципы**

1. **Group-first, not account-first.** Базовая единица — не счёт пользователя, а группа. Счёт — это техническая деталь. Группа — это продукт.

2. **Context-rich actions.** Каждое финансовое действие несёт контекст: кто, кому, зачем, в рамках чего. Контекст — это то, что превращает транзакцию в социальный объект.

3. **Shared view ≠ shared account.** Не нужен совместный банковский счёт (юридическая сложность). Нужна совместная витрина поверх индивидуальных счетов. Как Figma — каждый рисует в своём слое, но все видят общий холст.

4. **Invitation-as-action.** Приглашение встроено в core interaction. «Раздели счёт» = «пригласи участника». Невозможно использовать без других.

5. **Multi-group membership.** Один юзер = множество групп. Архитектура должна поддерживать это нативно. Это механизм кросс-кластерного распространения.

6. **Accumulating value.** Каждое действие оставляет след. История, паттерны, прогресс целей — всё это накапливается и увеличивает стоимость переключения.

## ЧАСТЬ II. КЕЙС — СБОРКА В КОНКРЕТНЫЙ ПРОДУКТ

### 11. Выбор сегментов

Формат: 4 сегмента с обоснованием выбора

Критерии отбора сегментов:

- Существует регулярное финансовое взаимодействие между людьми
- Текущее решение — примитивное (peer → bank → peer, n-1-n)
- Есть естественная группа (не нужно создавать с нуля)
- Высокая частота взаимодействий (не раз в год)
- Потенциал прямого + непрямого NE

Выбранные сегменты:

| # | Сегмент | Размер группы | Частота финансового взаимодействия | Текущее решение | Боль |
|---|---------|--------------|-----------------------------------|----------------|------|
| 1 | Пары (совместный быт) | 2 | Ежедневно | Переводы по Сбербанку + Excel / «ну примерно пополам» | Нет прозрачности, неловкость обсуждения, «кто больше тратит» |
| 2 | Друзья в регулярных активностях (бары, рестораны, поездки) | 3–8 | 2–4 раза в месяц | Один платит картой → потом «скинь мне» в мессенджере → кто-то забывает | Неловкость напоминания, потеря денег, хаос в чатах |
| 3 | Семьи с зависимыми членами (дети, пожилые родители) | 3–7 | Ежедневно | Переводы на карту ребёнку + звонок «у бабушки закончились деньги» | Нет контроля, нет видимости, нет общей картины |
| 4 | Соседи / коммунальные группы (совместная аренда, коммуналка) | 2–5 | Ежемесячно | Один платит → рассылает фото квитанции → «скиньте по 3500» | Один человек всегда «бухгалтер», остальные забывают |

Обоснование выбора:

- Все 4 сегмента имеют естественный социальный граф (не нужно создавать)
- Все 4 имеют регулярные финансовые взаимодействия (не разовые)
- Все 4 страдают от паттерна n-1-n (деньги идут через одного человека / через банк как «тупую трубу»)
- Все 4 имеют потенциал для прямого NE (ценность группы растёт с каждым участником)
- Сегменты пересекаются: один человек может быть в паре (сегмент 1) + ходить с друзьями в бар (сегмент 2) + помогать родителям (сегмент 3) + снимать квартиру с соседом (сегмент 4) → мультигрупповое участие = кросс-кластерное распространение

### 12. Текущая потребность

Формат: для каждого сегмента — потребность (явная / скрытая) + текущий путь + боль

#### Сегмент 1: Пары

Явная потребность: «Мы хотим понимать, кто сколько тратит и справедливо ли делим расходы»

Скрытая потребность (которую нужно сформировать): «Мы хотим чувствовать себя командой в вопросе денег, а не двумя отдельными людьми с конфликтом интересов»

Текущий путь:

1. Один платит за ужин / продукты / квартиру
2. Второй переводит «примерно половину» через банковское приложение
3. Иногда ведут Excel / заметки в телефоне / Splitwise
4. Раз в месяц (или никогда) «сверяются» — часто с раздражением
5. Возникают скрытые обиды: «я всегда плачу за продукты», «ты тратишь больше на себя»

Почему неудобно:

- Нет единой картины — каждый видит только свои траты
- Перевод через банк — это «тупая труба», не несёт контекста
- Excel/заметки — требуют дисциплины, забрасываются через 2 недели
- Splitwise — хорош для долгов, но не для совместной финансовой картины
- Разговор о деньгах = конфликт. Нет нейтрального «медиатора»

Проверка NE: текущие решения не сетевые. Excel — для одного. Банковский перевод — point-to-point, без контекста. Splitwise — ближе всего, но только фиксирует долги, не создаёт совместную финансовую реальность.

---

#### Сегмент 2: Друзья в регулярных активностях

Явная потребность: «Хочу легко разделить счёт и чтобы все вернули деньги без напоминаний»

Скрытая потребность: «Хочу, чтобы деньги не портили дружбу. Чтобы не быть "тем парнем, который напоминает про долг". Чтобы совместные траты были частью кайфа, а не головной болью после»

Текущий путь:

1. Группа идёт в ресторан / бар / поездку
2. Один человек платит за всех (или каждый пытается заплатить свою часть — хаос с официантом)
3. Плативший пишет в групповой чат: «Ребят, ужин был 8000, с каждого по 2000»
4. Двое переводят сразу. Один «забывает». Один считает, что он пил меньше и должен 1500
5. Неловкие напоминания. Или молчание и обида.
6. Через 3 дня кто-то переводит. Через неделю — ещё один. Четвёртый «забыл навсегда».

Почему неудобно:

- Один человек всегда берёт на себя роль «кассира» — это социально неприятно
- Напоминание о долге = микроконфликт
- Нет истории: «ты мне в прошлый раз не вернул» — невозможно доказать
- Деньги в мессенджере смешиваются с обычной перепиской — теряются
- Чем больше группа — тем хаотичнее

Проверка NE: боль масштабируется с размером группы. Это идеальный кейс для прямого NE: чем больше участников в системе — тем меньше хаоса. И для непрямого NE: один юзер создаёт контекст (счёт, событие) — остальные приходят за контекстом (увидеть долг, оплатить).

---

#### Сегмент 3: Семьи с зависимыми членами

Явная потребность: «Хочу дать ребёнку/родителю деньги и понимать, на что они тратятся»

Скрытая потребность: «Хочу заботиться финансово, не контролируя. Хочу видеть, что бабушка в порядке, без звонка "у тебя есть деньги?". Хочу, чтобы ребёнок учился обращаться с деньгами, но в безопасной среде»

Текущий путь:

1. Родитель переводит ребёнку на карту фиксированную сумму
2. Ребёнок тратит — родитель не видит на что (или видит выписку, но без контекста)
3. Ребёнок просит ещё — конфликт «куда дел?»
4. С пожилыми родителями: «Мам, тебе деньги нужны?» → «Нет, всё хорошо» (хотя закончились)
5. Несколько родственников помогают бабушке — не координируют между собой → кто-то переводит дважды, кто-то думает «уже другой перевёл»

Почему неудобно:

- Нет совместной видимости: мама и папа не видят общую картину расходов ребёнка
- Нет координации: тётя, дядя и родители помогают бабушке несинхронно
- Контроль ≠ забота — нет инструмента для «мягкого» участия
- Банковские «детские карты» — это счёт с лимитом, не совместный финансовый инструмент

Проверка NE:

- Прямой: чем больше членов семьи подключено → тем полнее картина, тем лучше координация
- Непрямой: один член семьи (мама) настраивает структуру → другие (папа, бабушка, ребёнок) получают готовый контекст
- Новый → старый: когда бабушка подключается, мама видит её баланс и перестаёт волноваться = ценность для мамы растёт

---

#### Сегмент 4: Соседи / коммунальные группы

Явная потребность: «Хочу, чтобы все вовремя скинулись на коммуналку / аренду и я не был бухгалтером»

Скрытая потребность: «Хочу жить с соседями без финансового напряжения. Чтобы деньги были "решённым

вопросом", а не ежемесячной драмой»

Текущий путь:

1. Один сосед платит за всё (аренда, коммуналка, интернет, уборка)
2. Фотографирует квитанцию → скидывает в чат
3. «Ребят, скиньте по 4200 до пятницы»
4. Кто-то скидывает сразу. Кто-то через неделю. Кто-то «забывает»
5. «Бухгалтер» чувствует себя в ловушке: все жили, а он считал и напоминал
6. При выезде одного соседа — хаос с перерасчётом

Почему неудобно:

- Роль «бухгалтера» — непропорциональная нагрузка на одного
- Напоминания = портят отношения
- Нет прозрачности: «а почему в этом месяце больше?»
- При смене состава — ручной перерасчёт
- Квитанции в чате теряются

Проверка NE:

- Прямой: ценность системы = f(количество подключённых соседей). Если один не подключён — дыра в системе
- Непрямой: «бухгалтер» создаёт контекст (загружает квитанцию, распределяет суммы) → остальные приходят за контекстом (видят сколько, за что, до когда)
- Новый → старый: когда последний сосед подключается — «бухгалтер» наконец освобождается от роли → ценность для него максимальна

### 13. Предложенное решение

Формат: общая концепция + адаптация под каждый сегмент + архитектура взаимодействия

#### 13.1. Общая концепция: «Финансовые группы» (рабочее название)

One-liner: Мессенджер для денег — где финансовые действия между людьми так же просты, как сообщения, и имеют контекст, историю и совместную видимость.

Суть: Не новый банк. Не кошелёк. Не калькулятор долгов. А слой совместных финансовых взаимодействий поверх существующих банковских счетов.

Ключевое отличие от существующих решений:

| Существующее | Что делает | Чего не хватает |
|-------------|-----------|----------------|
| Банковский перевод | Двигает деньги A → B | Нет контекста, нет группы, нет истории |
| Splitwise | Считает долги | Не двигает деньги, нет совместной картины |
| Совместный счёт | Общий пул денег | Юридическая сложность, негибко, «все или ничего» |
| Excel / заметки | Ручной учёт | Нет автоматизации, забрасывается, не социальный |

Наше решение:

- Группа — центральная сущность (не счёт, не юзер)
- Контекст — каждое финансовое действие привязано к ситуации (ужин, аренда, подарок маме, поездка)
- Shared view — все участники видят одну и ту же картину (кто сколько потратил, кто кому должен, как идёт общая цель)
- Действия, невозможные в одиночку — split, общая цель, координированная помощь, ритуал

Проверка через NE:

- Можно ли использовать в одиночку? — Нет. Группа из одного бессмысленна.
- Растёт ли ценность от новых участников? — Да. Каждый новый = полнее картина + меньше хаоса.
- Встроено ли приглашение в core action? — Да. «Раздели счёт» = «добавь участников» = приглашение.

#### 13.2. Единое решение с адаптацией под сегменты

Архитектура одна. Шаблоны (templates) — разные.

| Сегмент | Шаблон группы | Core interaction | Специфичные фичи |
|---------|--------------|-----------------|-------------------|
| Пары | «Мы вдвоём» | Добавление расхода в общий контекст → автоматический расчёт баланса | Общий бюджет с категориями, мягкие лимиты, «кто сколько внёс в этом месяце», без формализации долга — скорее дашборд отношений |
| Друзья | «Наша компания» | Split расхода → уведомление участникам → оплата в один тап | Привязка к событиям (ужин, поездка, бар), история ивентов, «кто вечный должник» (с юмором), автоматические напоминания от системы (не от человека) |
| Семья | «Наша семья» | Перевод с контекстом + видимость для всех координирующих | Роли (родитель / ребёнок / опекаемый), лимиты, «тихий мониторинг» (вижу баланс бабушки без её активного участия), координация помощи |
| Соседи | «Наша квартира» | Загрузка расхода (квитанция) → автоматическое распределение → оплата | Регулярные расходы, автоплатежи, ротация «бухгалтера», прозрачная история, лёгкое добавление/удаление участника |

Почему одно решение, а не четыре:

- Один юзер — в нескольких группах → мультигрупповое участие → кросс-кластерное распространение
- Базовые примитивы одинаковые: группа, контекст, shared view, действие
- Разница — в шаблонах и дефолтных настройках, не в архитектуре
- Снижается стоимость разработки и поддержки

### 14. Ценность и изменение поведения

Формат: для каждого сегмента — почему ценно + почему поменяют привычки + почему приведут друзей

#### 14.1. Почему ценно (по сегментам)

| Сегмент | Текущая боль | Наша ценность | В чём изменение |
|---------|-------------|---------------|----------------|
| Пары | Деньги = источник конфликтов | Деньги = совместный проект, прозрачный и спокойный | От «кто кому должен» к «мы вместе видим и решаем» |
| Друзья | Один платит → неловко просить → теряет деньги | Система просит за тебя, нейтрально и автоматически | От «скинь мне» к «система уже посчитала, нажми кнопку» |
| Семья | Помощь вслепую, нет координации | Вижу, что нужно, координирую с другими, забочусь без навязчивости | От «мам, у тебя есть деньги?» к «вижу, что баланс низкий — перевожу» |
| Соседи | Один «бухгалтер» страдает, остальные забывают | Система — автоматический бухгалтер, который никому не портит отношения | От «скиньте по 3500» к «квитанция загружена, нажмите оплатить» |

#### 14.2. Почему поменяют привычки

Триггер изменения поведения — модель Фогга (Behavior = Motivation × Ability × Trigger):

| Компонент | Как обеспечиваем |
|-----------|-----------------|
| Motivation | Убираем социальное трение (не нужно просить, напоминать, конфликтовать). Мотивация — не «хочу вести бюджет», а «хочу нормальные отношения с людьми, с которыми делю деньги» |
| Ability | Минимальный барьер входа: не нужно регистрироваться в новом банке, не нужно переводить деньги на новый счёт. Работает поверх существующих карт и счетов. Первый опыт — 30 секунд: «тебе пришёл запрос на 500₽ за ужин — оплати в один тап». Новый юзер попадает не в пустое приложение, а в готовый контекст (непрямой NE от старого юзера). |
| Trigger | Встроен в core action. Не push-уведомление «Открой приложение». А: «Маша разделила счёт за ужин — тебе 1200₽». Триггер = действие другого человека. Это и есть сетевой эффект в чистом виде: активность одного юзера создаёт триггер для другого. |

**Почему переключение произойдёт:**

- Не просим заменить банк — добавляем слой сверху
- Первое действие = ответ на запрос от друга (нулевое усилие инициации)
- Ценность мгновенная: я сразу вижу, сколько должен, за что, кому — и закрываю в один тап
- Второй раз пользуюсь — когда сам плачу за всех и хочу разделить (теперь я инициатор — я создаю контекст → непрямой NE)

#### 14.3. Почему приведут друзей

**Ключевой тест: «Приглашение — это жертва или выгода?»**

В большинстве продуктов приглашение = жертва: «потрать своё время и социальный капитал, чтобы помочь компании расти». Пользователь — бесплатный маркетолог.

**В нашем продукте приглашение = выгода для приглашающего:**

| Сегмент | Почему юзер сам хочет пригласить | Что получает от приглашения |
|---------|----------------------------------|---------------------------|
| **Пары** | «Хочу, чтобы партнёр тоже видел общую картину — тогда не будет конфликтов» | Прозрачность, спокойствие, общее понимание |
| **Друзья** | «Хочу, чтобы все были в системе — тогда мне не нужно никому писать "скинь"» | Избавление от роли «кассира», автоматический возврат денег |
| **Семья** | «Хочу, чтобы бабушка была подключена — тогда я вижу, что с ней всё ок» | Спокойствие, координация с другими родственниками |
| **Соседи** | «Хочу, чтобы все соседи были в системе — тогда я перестану быть бухгалтером» | Освобождение от роли, автоматизация, меньше конфликтов |

**Механика приглашения встроена в продукт, а не параллельна ему:**

Юзер совершает core action (split / общая цель / загрузка квитанции)
         │
         ▼
Система: "Добавь участников"
         │
         ├── Участник уже в системе → мгновенное уведомление
         │
         └── Участник НЕ в системе → приглашение с готовым контекстом
              │
              ▼
         "Маша разделила ужин в Кофемании. Тебе — 1200₽.
          Нажми, чтобы оплатить."
              │
              ▼
         Новый юзер попадает в КОНТЕКСТ, а не в пустое приложение
         (непрямой NE: старый юзер создал контекст → новый получает ценность)

**Виральная петля:**

1. Юзер А платит за ужин → делит счёт через продукт
2. Юзеры B, C, D получают приглашение с контекстом → регистрируются → платят
3. Через неделю юзер B платит за бар → делит счёт → приглашает юзеров E, F (которых не было в первом ужине)
4. Юзер B — мост между кластерами (кросс-кластерное распространение)
5. Юзер E попадает в систему → добавляет в другую группу (семью) → новый кластер

**Ключевое:** каждый split = органическое приглашение 2–7 человек. Это не маркетинг. Это архитектура продукта.

### 15. Архитектура / дизайн

**Формат:** прототип в нескольких представлениях для разных аудиторий

#### 15.1. Для продактов — user story map

**Эпик 1: Создание группы**

Как [юзер], я хочу [создать финансовую группу], чтобы [совместно управлять расходами с конкретными людьми]

Истории:

- Создать группу из контактов телефона
- Выбрать шаблон (пара / друзья / семья / соседи / кастомный)
- Пригласить участников через ссылку / SMS / мессенджер
- Принять приглашение и попасть в готовый контекст

**Эпик 2: Core interaction — Split**

Как [юзер-плательщик], я хочу [разделить расход между участниками группы], чтобы [каждый знал, сколько должен, и мог оплатить в один тап]

Истории:

- Добавить расход (сумма + описание + фото чека опционально)

- Выбрать участников и способ деления (поровну / по суммам / по процентам)
- Привязать к контексту (событие / место / категория)
- Отправить запросы на оплату участникам
- Получить уведомление о запросе → оплатить в один тап
- Видеть статус: кто оплатил, кто нет

**Эпик 3: Shared View**

Как [участник группы], я хочу [видеть общую финансовую картину группы], чтобы [понимать баланс, историю и тренды]

Истории:

- Дашборд группы: кто кому сколько должен (net balance)
- История всех расходов с контекстом
- Аналитика: сколько потратили за месяц, на что, кто больше
- Прогресс общей цели (если есть)

**Эпик 4: Общие цели**

Как [участник группы], я хочу [создать общую финансовую цель], чтобы [вместе копить на что-то]

Истории:

- Создать цель (название + сумма + дедлайн)
- Каждый вносит свою часть в удобном темпе
- Прогресс-бар виден всем
- Уведомления о вкладах других (социальное давление / мотивация)
- Celebration при достижении цели

**Эпик 5: Ритуалы и автоматизация**

Как [участник группы], я хочу [настроить регулярные финансовые действия], чтобы [не делать одно и то же вручную каждый месяц]

Истории:

- Регулярный split (ежемесячная аренда, коммуналка)
- Автонапоминания (система напоминает, не человек)
- Еженедельный / ежемесячный ритуал (family pool, savings challenge)

**Ключевые архитектурные решения для IT:**

| Решение | Обоснование через NE |
|---------|---------------------|
| Event-driven архитектура | Каждое действие юзера порождает события для других юзеров — это и есть механика NE. Event bus = артерия сетевого эффекта |
| Group как first-class entity | Группа — не атрибут юзера, а центральная сущность. Все действия происходят в контексте группы |
| Contextual invite landing | Новый юзер приходит не в пустой онбординг, а в конкретный контекст (долг, цель, группа). Это обеспечивает мгновенную ценность от непрямого NE |
| Materialized views для Shared View | Дашборд группы должен обновляться в реальном времени — каждый видит последствия действий других. Это делает NE осязаемым |
| Multi-group membership на уровне ядра | Один юзер = много групп. Не «переключение аккаунтов», а нативная навигация. Обеспечивает кросс-кластерное распространение |
| Idempotent payment requests | Защита от дублирования запросов — критично для доверия в финансовых группах |

### 16. Свобода использования

**Формат:** анализ гибкости системы + потенциальное неожиданное поведение

#### 16.1. Заложенная гибкость

**Принцип: «Лего, а не конструктор IKEA»**

Мы даём примитивы (группа, split, цель, контекст, ритуал) — юзеры собирают из них что хотят. Шаблоны (пара, друзья, семья, соседи) — это лишь стартовые пресеты, а не ограничения.

| Примитив | Задуманное использование | Возможное «не по инструкции» |
|---------|-------------------------|------------------------------|
| **Группа** | Пара, семья, друзья, соседи | Рабочая команда; клуб по интересам; группа анонимных транжир; репетитор + ученики |
| **Split** | Разделить ресторанный счёт | Собрать деньги на благотворительность; скинуться на чаевые курьеру; разделить стоимость подписки Netflix между 5 аккаунтами |
| **Общая цель** | Накопить на поездку | Челлендж «кто больше сэкономит»; фонд на чрезвычайные ситуации группы; копилка на новогодний корпоратив |
| **Контекст** | Привязка к ужину / квитанции | Дневник совместных трат как «летопись отношений»; геймификация («мы потратили на кофе больше, чем на спорт») |
| **Ритуал** | Ежемесячная коммуналка | Еженедельные «финансовые пятиминутки» пары; ежедневный savings challenge; регулярные взносы в фонд помощи родственнику |

#### 16.2. Потенциальное неожиданное поведение (emergent use cases)

**Именно здесь прячутся настоящие сетевые эффекты — когда юзеры изобретают то, что мы не планировали.**

**Сценарий 1: «Микробизнес на коленке»**

- Репетитор создаёт группу с учениками → выставляет ежемесячные запросы → ученики платят в один тап
- Мы не проектировали инвойсинг — но примитивы позволяют
- NE: чем больше учеников в системе, тем удобнее репетитору; каждый ученик видит, что это удобно → рекомендует своему репетитору по другому предмету

**Сценарий 2: «Социальное давление на сбережения»**

- Группа друзей создаёт цель «Не потратить на delivery больше 10к в месяц»
- Каждый видит прогресс других → стыдно быть аутсайдером
- Мы не проектировали финансовый фитнес — но механика целей + группа + видимость = Strava для денег

**Сценарий 3: «Родительский комитет»**

- Родители в школе создают группу → скидываются на подарок учителю / ремонт класса / экскурсию
- Казначей загружает чеки → все видят, на что потрачено → прозрачность
- Мы не проектировали для школ — но группа + split + контекст = готовое решение

**Сценарий 4: «Кредит доверия» между друзьями**

- Один друг одалживает другому через систему → оба видят долг → нет неловкости
- Со временем формируется «кредитная история дружбы»: кто надёжный, кто задерживает
- Мы не проектировали P2P-кредитование — но история + группа + баланс = неявный кредитный механизм

**Сценарий 5: «Комьюнити-фонд»**

- Соседи по дому создают группу → общий фонд на двор / детскую площадку / охрану
- Ежемесячные автоматические взносы → прозрачный баланс → голосование на что потратить
- Мы не проектировали ТСЖ — но ритуал + цель + группа + shared view = готовое решение

#### 16.3. Почему свобода усиливает NE

Больше свободы использования
         │
         ▼
Юзеры изобретают новые кейсы
         │
         ▼
Каждый новый кейс = новый тип группы
         │
         ▼
Один юзер участвует в ещё большем числе групп
         │
         ▼
Больше мультигруппового участия = больше кросс-кластерного распространения
         │
         ▼
Сильнее NE

**Архитектурный вывод:** НЕ хардкодить сценарии. Шаблоны — только как starting point. Примитивы должны быть достаточно гибкими, чтобы юзеры собирали неожиданные конфигурации. Наша задача — наблюдать за emergent behavior и превращать самые успешные паттерны в новые шаблоны.

### 17. Engagement

**Формат:** модель вовлечения, привязанная к NE

#### 17.1. Engagement-модель: почему юзеры возвращаются

**Ключевой принцип:** в сетевом продукте engagement обеспечивают другие люди, а не фичи приложения.

| Что возвращает юзера | Тип NE | Механика | Частота |
|---------------------|--------|----------|---------|
| «Тебе пришёл запрос на оплату» | Непрямой (контекстный) | Действие другого юзера = триггер для меня | По событию (2–8 раз в месяц) |
| «Катя внесла 3000₽ в общую цель» | Прямой (групповой) | Активность участника = социальное давление / мотивация | По событию |
| «Ваш ежемесячный отчёт: группа "Квартира" потратила 45к» | Непрямой (данные) | Накопленные данные группы = инсайт | 1 раз в месяц |
| «Время скинуться на аренду» | Ритуал | Автоматический триггер привычного действия | Регулярно (неделя/месяц) |
| «В группе "Девичник" новый расход» | Прямой (групповой) | Лента активности группы | По событию |
| «Ты уже 3 месяца в группе — вот ваша статистика» | Непрямой (данные) | Накопленная ценность становится видимой | Milestone-based |

#### 17.2. Engagement-петли (loops)

**Петля 1: Транзакционная (короткая, высокочастотная)**

Юзер А платит за что-то
    │
    ▼
Юзер А делит через продукт (core action)
    │
    ▼
Юзеры B, C получают уведомление (NE-триггер)
    │
    ▼
Юзеры B, C оплачивают (engagement)
    │
    ▼
Юзер А видит "оплачено" (reward)
    │
    ▼
В следующий раз юзер B или C платит и делит (цикл повторяется с новым инициатором)

**Время цикла:** часы — дни
**NE-тип:** непрямой контекстный (инициатор создаёт контекст → участники приходят за контекстом) + прямой групповой (чем больше участников, тем удобнее)

**Петля 2: Аналитическая (средняя, ежемесячная)**

Группа совершает транзакции в течение месяца
    │
    ▼
Система формирует аналитику (кто сколько потратил, на что, тренды)
    │
    ▼
Участники получают monthly digest ("Ваша группа 'Квартира' в марте")
    │
    ▼
Участники открывают → видят инсайты → обсуждают
    │
    ▼
Обсуждение → решения ("давайте в следующем месяце меньше на доставку")
    │
    ▼
Решения → новые цели / лимиты / ритуалы в продукте
    │
    ▼
Больше действий в продукте → богаче аналитика следующего месяца

**Время цикла:** месяц
**NE-тип:** непрямой (данные) — каждая транзакция каждого юзера обогащает общую аналитику для всех. Чем больше участников и чем дольше используют — тем ценнее.
**Критично:** аналитика одного юзера бесполезна. Аналитика группы — ценна. Это чистый NE.

**Петля 3: Социальная / мотивационная (средняя, ongoing)**

Юзер А вносит вклад в общую цель / выполняет savings challenge
    │
    ▼
Остальные участники видят уведомление ("Маша отложила 5000₽!")
    │
    ▼
Социальное давление + мотивация → другие тоже вносят
    │
    ▼
Прогресс-бар двигается → дофаминовый reward для всех
    │
    ▼
Чем ближе к цели → тем сильнее мотивация → ускорение вкладов
    │
    ▼
Цель достигнута → celebration → новая цель

**Время цикла:** дни — недели
**NE-тип:** прямой групповой — активность одного участника мотивирует других. Буквально: ценность продукта для юзера Б растёт из-за активности юзера А.
**Аналогия:** Strava — ты бегаешь больше, потому что видишь, что друг пробежал 10 км.

**Петля 4: Ритуальная (длинная, формирующая привычку)**

Группа настраивает ритуал (ежемесячная коммуналка / еженедельный family pool)
    │
    ▼
Система автоматически создаёт split / запрос в назначенное время
    │
    ▼
Участники получают привычный триггер → привычное действие → привычный reward
    │
    ▼
Со временем: действие становится автоматическим (habit loop)
    │
    ▼
Ритуал = якорь, который удерживает группу в продукте
    │
    ▼
Чем дольше ритуал → тем выше switching cost (нужно воссоздать привычку заново)

**Время цикла:** неделя — месяц (регулярный)
**NE-тип:** прямой групповой — ритуал имеет смысл только с группой. Один человек не может выполнить ритуал «скинуться на коммуналку».
**Критично:** ритуалы — самый мощный retention-механизм. Транзакционная петля может прерваться (нет повода). Ритуальная — работает по расписанию.

#### 17.3. Engagement через жизненный цикл юзера

| Фаза | Что происходит | Какой NE работает | Ключевая метрика |
|------|---------------|-------------------|-----------------|
| **День 0: Приглашение** | Юзер получает ссылку с контекстом («тебе 1200₽ за ужин») | Непрямой: старый юзер создал контекст → новый приходит за ним | Conversion rate приглашения → регистрация |
| **День 0: Первое действие** | Оплачивает свою часть / видит группу | Непрямой: мгновенная ценность от контекста, созданного другим | Time-to-value (секунды до первого полезного действия) |
| **Неделя 1: Второе использование** | Сам инициирует split / добавляет расход | Переход от потребителя NE к создателю NE | % юзеров, совершивших второе действие |
| **Неделя 2–4: Формирование привычки** | Несколько splits, появляется история, видит аналитику | Непрямой (данные): накопленная история = ценность | Retention D7, D14, D30 |
| **Месяц 1–3: Расширение** | Создаёт вторую группу (другой сегмент) | Кросс-кластерный: юзер = мост между группами | Кол-во групп на юзера |
| **Месяц 3+: Привычка** | Ритуалы, регулярные splits, checking dashboard | Прямой + непрямой + ритуальный | Monthly active groups, retention M3+ |
| **Месяц 6+: Lock-in** | Накопленная история, паттерны, привычки группы | Data NE + social NE | Churn rate (должен стремиться к нулю для активных групп) |

#### 17.4. Anti-engagement: что убьёт вовлечение

| Риск | Почему опасно | Как предотвращаем |
|------|--------------|-------------------|
| Спам уведомлениями | Юзер отключает нотификации → разрывается NE-петля | Умные дефолты: уведомления только о действиях, которые требуют ответа. Дайджесты вместо потока. |
| «Мёртвая» группа | Группа создана, но никто не использует → разочарование | Онбординг-подсказки: «Добавьте первый расход». Автоудаление пустых групп через 30 дней. |
| Один участник не платит | Все ждут → фрустрация → виноват продукт | Автонапоминания с эскалацией. «Settle up» — предложение закрыть мелкие долги. Опция «простить долг». |
| Неравномерная активность | Один юзер «тащит» группу, остальные пассивны | Ротация ролей: система предлагает «В этот раз Катя загружает квитанцию». Распределение нагрузки. |
| Финансовый конфликт в группе | Прозрачность обнажает проблему: «ты тратишь слишком много» | Деликатная аналитика: показывать факты без оценок. Опция скрыть детали для чувствительных категорий. |
| Приватность | «Не хочу, чтобы все видели мои расходы» | Shared view показывает только то, что юзер явно добавил в группу. Личные расходы вне группы — невидимы. Полный контроль: что расшарить, а что нет. |

#### 17.5. Метрики engagement, привязанные к NE

Не просто DAU/MAU — а метрики, которые измеряют силу сетевого эффекта:

| Метрика | Что измеряет | Как связана с NE | Целевое значение |
|---------|-------------|-----------------|-----------------|
| Groups per user | Кол-во групп, в которых состоит юзер | Мультигрупповое участие = кросс-кластерное распространение | >2 через 3 месяца |
| Group density | % участников группы, активных за месяц | Здоровье прямого NE: все ли участники «живы» | >80% |
| Organic invite rate | % новых юзеров, пришедших через core action (не маркетинг) | Чистый NE-рост vs купленный рост | >60% |
| Contextual conversion | % людей, получивших invite с контекстом → зарегистрировались | Сила непрямого NE (контекст от старого юзера) | >40% |
| Creator/consumer ratio | % юзеров, которые сами инициировали split (создатели контекста) vs только оплачивали | Здоровье непрямого NE: достаточно ли «создателей»? | >50% creators через месяц |
| Time-to-second-group | Время от регистрации до создания/вступления во вторую группу | Скорость кросс-кластерного распространения | <30 дней |
| Ritual retention | % групп, сохраняющих регулярный ритуал через 3 месяца | Сила ритуальной петли | >60% |
| Net balance settlement rate | % долгов, закрытых в течение 48 часов | Здоровье транзакционной петли | >70% |
| Network value score | Совокупная оценка: «Насколько ценнее стал продукт для юзера X за последний месяц из-за действий других юзеров?» | Прямое измерение NE | Растёт month-over-month |

### 18. Monetization (бонусная секция)

Формат: модель монетизации, совместимая с NE

Принцип: монетизация не должна разрушать NE. Если берём деньги за то, что усиливает сеть — мы тормозим собственный рост.

| Модель | Описание | Влияние на NE | Оценка |
|--------|---------|---------------|--------|
| Freemium по группам | Бесплатно: до 3 групп, до 5 участников. Платно: безлимит + расширенная аналитика | Не ломает NE: core interaction бесплатен. Лимиты мотивируют мультигрупповых юзеров платить. | ✅ Рекомендуется |
| Interchange / комиссия с платежей | Микрокомиссия с каждой оплаты через систему (0.5–1%) | Риск: юзеры будут платить вне системы, чтобы избежать комиссии → разрушает NE | ⚠️ Осторожно, только если ценность сильно превышает комиссию |
| Premium-аналитика | Продвинутые инсайты: прогнозы, паттерны, сравнение с похожими группами | Усиливает NE: аналитика тем ценнее, чем больше данных от группы | ✅ Хорошо |
| Групповая покупательная способность (marketplace) | Группы получают скидки на сервисы (подписки, страховки, доставка) за счёт объёма | Прямой NE: чем больше группа → лучше скидка. Мотивирует приглашать. | ✅✅ Идеально — монетизация усиливает NE |
| White-label / B2B | Банки / финтехи встраивают наши «финансовые группы» в свои приложения | Расширяет NE на уровне инфраструктуры: группы работают кросс-банковски | ✅ Стратегически |

### 19. Проверка потребности — как будем валидировать

Формат: план эксперимента в 3 волны

#### Волна 1: Качественная (неделя 1–3)

Цель: проверить, существует ли боль и резонирует ли решение

| Метод | Кол-во | Сегмент | Что проверяем |
|-------|--------|---------|--------------|
| Глубинные интервью | 5–7 на сегмент (20–28 всего) | Все 4 сегмента | «Расскажите про последний раз, когда вы делили деньги с кем-то. Что было неудобно?» |
| Дневниковое исследование (5 дней) | 10 участников | Пары + друзья | «Записывайте каждый раз, когда возникает ситуация совместных денег. Что делали? Что чувствовали?» |
| Анализ чатов (с разрешения) | 10 групповых чатов | Друзья + соседи | Как часто появляются сообщения «скинь мне» / фото квитанций / обсуждение долгов |

Ключевые вопросы для интервью:

- Как часто вы делите расходы с кем-то?
- Кто обычно платит? Как решаете?
- Были ли ситуации, когда деньги портили отношения?
- Что используете сейчас? Что бесит?
- Если бы я показал вам [концепт] — стали бы пользоваться? Почему? С кем?
- Кого бы вы первым позвали в такой продукт? Почему?

Criteria pass/fail:

- ✅ Pass: >70% респондентов описывают боль спонтанно (до подсказки)
- ✅ Pass: >50% говорят «я бы точно позвал X» (конкретный человек, не абстрактный)
- ❌ Fail: боль существует, но низкочастотная (раз в полгода) — NE не запустится

#### Волна 2: Концептуальная (неделя 3–5)

Цель: проверить, работает ли core interaction и встроенное приглашение

| Метод | Что делаем | Что проверяем |
|-------|-----------|--------------|
| Кликабельный прототип (Figma) | Сценарий «split ужина» от начала до оплаты + приглашение нового юзера | Понятность, time-to-value, желание пригласить |
| «Wizard of Oz» в WhatsApp | Создаём бота / вручную имитируем продукт в реальной группе друзей на протяжении 2 недель. Человек-оператор выполняет роль системы: считает split, рассылает запросы, ведёт баланс | Реальное поведение: будут ли использовать? Как часто? Будут ли звать новых людей? Что будут делать «не по инструкции»? |
| Fake door test (landing) | Лендинг с описанием продукта + кнопка «Создать группу» / «Разделить первый счёт» → сбор email / номера телефона | Спрос: конверсия в «хочу попробовать». Тест заголовков / позиционирования по сегментам |
| Концепт-тест с парами | Показываем прототип «Совместный бюджет» 10 парам → просим обоих партнёров оценить независимо | Критично: оба ли партнёра видят ценность? Если только один — NE не сработает (нужны оба) |

**Criteria pass/fail:**

- ✅ Pass (прототип): >80% завершают сценарий split без подсказок; >60% говорят «хочу попробовать с реальными друзьями»
- ✅ Pass (Wizard of Oz): группа использует «систему» >3 раз за 2 недели; хотя бы 1 участник приглашает нового человека органически
- ✅ Pass (fake door): конверсия landing → email >15%
- ✅ Pass (пары): оба партнёра оценивают >7/10 по шкале «буду пользоваться»
- ❌ Fail: юзеры понимают концепцию, но не видят причины менять текущее поведение («и так нормально»)

#### Волна 3: Количественная + MVP (неделя 5–12)

**Цель:** проверить NE-механики на реальных данных с реальными деньгами

| Этап | Что делаем | Что измеряем |
|------|-----------|-------------|
| **MVP (неделя 5–8)** | Минимальный продукт: создание группы + split + оплата + приглашение с контекстом. Без аналитики, без целей, без ритуалов. Только core interaction. | Time-to-value, completion rate, organic invite rate |
| **Closed beta (неделя 8–10)** | Запуск на 50–100 «зерновых» групп (приглашаем конкретные компании друзей, пары, соседские группы). Каждому сегменту — свой шаблон. | Все NE-метрики из п.17.5 |
| **Наблюдение за emergent behavior (неделя 8–12)** | Не трогаем продукт, только наблюдаем. Что делают «не по инструкции»? Какие группы создают? Какие паттерны возникают? | Неожиданные use cases, feature requests, «хаки» |
| **Когортный анализ NE (неделя 10–12)** | Сравниваем когорты: группы с 2 участниками vs 4 vs 6+. Растёт ли engagement с размером группы? | Прямое измерение: ценность = f(размер группы)? |

**Ключевые эксперименты внутри MVP:**

**Эксперимент 1: «Контекстное приглашение vs обычное»**

Группа A: приглашение = ссылка с контекстом
  ("Маша разделила ужин, тебе 1200₽")
Группа B: приглашение = обычная ссылка
  ("Маша приглашает тебя в приложение")

Метрика: conversion rate приглашение → регистрация
Гипотеза: Группа A > Группа B минимум в 2 раза
Почему: непрямой NE — контекст от старого юзера = мгновенная ценность для нового

**Эксперимент 2: «Система напоминает vs человек напоминает»**

Группа A: автонапоминание от системы через 24ч
  ("Напоминаем: 1200₽ за ужин в Кофемании")
Группа B: нет автонапоминания, юзер сам решает
  (может написать в чат или промолчать)

Метрика: settlement rate (% долгов, закрытых за 48ч)
         + NPS / satisfaction инициатора split
Гипотеза: Группа A — settlement выше, satisfaction инициатора выше
Почему: убираем социальное трение — главный барьер текущего поведения

**Эксперимент 3: «Мультигрупповое участие и распространение»**

Наблюдаем: какой % юзеров, пришедших через приглашение в группу X, создают или вступают в группу Y (другой сегмент) в течение 30 дней

Метрика: cross-cluster rate
Гипотеза: >20% юзеров за 30 дней попадают во вторую группу
Почему: мультигрупповое участие — ключевой механизм масштабирования NE

**Эксперимент 4: «NE существует — прямое измерение»**

Сравниваем:

- Retention юзеров в группах из 2 человек
- Retention юзеров в группах из 4+ человек

Гипотеза: retention в группах 4+ значимо выше
Почему: если да — это прямое доказательство NE: ценность для юзера растёт от количества других юзеров.

Дополнительно:

- Retention юзера в 1 группе vs юзера в 3+ группах
- Гипотеза: 3+ групп → retention значимо выше

**Criteria pass/fail для MVP:**

- ✅ Pass: organic invite rate >40% (больше половины новых юзеров приходят через core action, не маркетинг)
- ✅ Pass: groups per user >1.5 через 30 дней (мультигрупповое участие работает)
- ✅ Pass: retention D30 в группах 4+ > retention D30 в группах из 2 (NE существует)
- ✅ Pass: settlement rate >60% за 48 часов (core interaction работает)
- ✅ Pass: хотя бы 3 emergent use cases, которые мы не проектировали (продукт достаточно гибкий)
- ❌ Fail: organic invite rate <20% (юзеры не приглашают → NE не запускается)
- ❌ Fail: нет разницы в retention между маленькими и большими группами (NE не существует)
- ❌ Fail: >50% групп «мертвы» через 2 недели (core interaction не цепляет)

### 21. Риски и митигация

Формат: таблица ключевых рисков с оценкой влияния на NE

| # | Риск | Вероятность | Влияние на NE | Митигация |
|---|------|------------|---------------|-----------|
| 1 | «Холодный старт»: не получается собрать первые группы с достаточной плотностью | Высокая | Критическое — без первых активных групп NE не запускается | Начинаем с hand-picked групп: конкретные компании друзей, конкретные пары. Не массовый запуск, а «посев» в плотные кластеры. Минимальный юнит = 2–3 человека, не нужна массовая критическая масса |
| 2 | Приватность: «не хочу, чтобы люди видели мои деньги» | Высокая | Среднее — может ограничить adoption | Shared view показывает только то, что юзер явно добавил. Вне группы — ничего не видно. Полный контроль. Позиционирование: «не прозрачность, а совместная видимость того, что вы и так обсуждаете в чате» |
| 3 | Регуляторика: лицензии на перевод денег, требования ЦБ, KYC/AML | Высокая | Среднее — может замедлить, но не убить | Фаза 1: работаем как слой поверх существующих платёжных систем (Stripe, банковские API). Мы не храним деньги, мы оркестрируем контекст. Деньги идут bank-to-bank. Позднее: собственная лицензия если нужно |
| 4 | Копирование банками: Сбер / Тинькофф запускают «группы» внутри своего приложения | Средняя | Высокое если до NE, низкое после NE | Наш moat: кросс-банковский (работаем поверх любых счетов). Банк может сделать группы только для своих клиентов → в реальной компании друзей у всех разные банки → банковское решение покрывает 30% группы, наше — 100%. Плюс: если NE запущен, переключение = перетаскивание всей группы + потеря истории |
| 5 | **«Split — это commodity»:** конкуренция с Splitwise, Tricount, Settle Up | Средняя | Среднее | Splitwise = калькулятор долгов, не двигает деньги. Мы = контекст + оплата + shared view + цели + ритуалы. Конкурируем не на уровне фичи «split», а на уровне «финансовые отношения между людьми». Splitwise — инструмент. Мы — среда |
| 6 | **Токсичная прозрачность:** продукт обнажает финансовое неравенство в группе, вызывает конфликты | Средняя | Высокое — юзеры уйдут и расскажут другим (анти-NE) | Деликатный дизайн: показываем факты без оценок и рейтингов. Нет «таблицы лидеров» по тратам. Опция скрыть суммы, показывать только % или категории. Тестируем на интервью: «что бы вы НЕ хотели, чтобы видели другие?» |
| 7 | **Усталость от финтеха:** «ещё одно приложение для денег, у меня уже 5» | Высокая | Среднее — барьер первой установки | Первый контакт — не «скачай приложение», а «нажми ссылку и оплати 1200₽». Можно оплатить без установки (web). Установка — только когда юзер хочет инициировать сам. Минимальный барьер входа |
| 8 | **Мошенничество:** фейковые splits, фишинг через приглашения | Средняя | Высокое — подрывает доверие ко всей системе | Верификация через контакты телефона. Split возможен только между взаимными контактами. Лимиты на суммы для новых юзеров. Подтверждение перед оплатой незнакомым. Fraud detection на уровне паттернов |
| 9 | **Культурный барьер:** «о деньгах не говорят» / «считаться — стыдно» | Высокая | Критическое — может заблокировать adoption целых сегментов | Позиционирование: не «контролируй деньги друзей», а «убери неловкость из совместных трат». Система считает за тебя → тебе не нужно поднимать тему. Тон: лёгкий, без занудства. Тестируем формулировки на интервью |
| 10 | **Группа разваливается → юзер уходит** | Средняя | Среднее | Мультигрупповое участие: даже если одна группа «умерла», юзер остаётся ради других. Чем больше групп → тем устойчивее. Цель: >2 групп на юзера к месяцу 3 |

### 22. Итоговое резюме — Executive Summary

**Формат:** 1 страница для стейкхолдеров

---

**NETWORK EFFECTS В ФИНТЕХЕ: ПРОДУКТОВО-АРХИТЕКТУРНОЕ ПРЕДЛОЖЕНИЕ**

**Проблема:**
Финансовые взаимодействия между людьми (пары, друзья, семьи, соседи) проходят по примитивной схеме: peer → bank → peer (n-1-n). Банк — «тупая труба», которая двигает деньги без контекста, без групповой логики, без совместной видимости. Результат: один человек берёт на себя роль «кассира», неловкие напоминания о долгах, потерянные деньги, конфликты из-за непрозрачности.

**Инсайт:**
Ни один существующий продукт не владеет социальным слоем финансовых взаимодействий между знакомыми людьми. Splitwise считает долги, но не двигает деньги. Банки двигают деньги, но не понимают контекста. Venmo добавил социальную ленту, но не пошёл дальше. Поле пусто.

**Решение:**
«Финансовые группы» — слой совместных финансовых взаимодействий поверх существующих банковских счетов. Не новый банк, а новый тип связи между людьми и их деньгами.

**Сетевой эффект (core thesis):**

> Ценность продукта для каждого юзера растёт из-за активности, наличия и вклада других юзеров.

| Тип NE | Как работает в нашем продукте |
|--------|------------------------------|
| **Прямой групповой** | Чем больше участников в группе → тем полнее картина, тем удобнее, тем меньше хаоса |
| **Непрямой контекстный** | Один юзер создаёт split с контекстом → другие приходят за контекстом (в том числе новые юзеры) |
| **Кросс-кластерный** | Один юзер в нескольких группах = мост → продукт распространяется между кластерами органически |

Ключевые архитектурные решения:

1. Group-first: группа — центральная сущность, не счёт
2. Invitation-as-action: приглашение встроено в core interaction, не параллельно ему
3. Contextual onboarding: новый юзер попадает в контекст, а не в пустое приложение
4. Cross-bank layer: работаем поверх любых банков → 100% покрытие группы

Целевые сегменты:

1. Пары (совместный быт) — ежедневная частота
2. Друзья (рестораны, поездки, бары) — 2–8 раз в месяц
3. Семьи с зависимыми членами — ежедневная частота
4. Соседи / коммунальные группы — ежемесячная частота

Как проверяем:

- Волна 1 (нед. 1–3): глубинные интервью + дневники → подтверждение боли
- Волна 2 (нед. 3–5): прототип + Wizard of Oz → подтверждение концепции
- Волна 3 (нед. 5–12): MVP + closed beta → подтверждение NE

Ключевой критерий успеха:
Organic invite rate > 40% — больше половины новых юзеров приходят через core action (split), а не через маркетинг. Это прямое доказательство работающего сетевого эффекта.

Ключевой критерий провала:
Нет разницы в retention между группами из 2 и 4+ человек. Это значит, что NE не существует и продукт — просто ещё один инструмент, а не сетевой продукт.

Монетизация:

- Фаза 1–2: freemium (core бесплатен, premium-аналитика платная)
- Фаза 3+: групповая покупательная способность (marketplace) — монетизация, которая усиливает NE

Защитный ров (moat) через 12+ месяцев:

- Накопленные данные групп (невоспроизводимы)
- Социальный граф финансовых отношений (уникален)
- Привычки и ритуалы (высокий switching cost)
- Кросс-банковская инфраструктура (банк не может повторить)

## ЧАСТЬ III. ПРИЛОЖЕНИЯ

### Приложение A: Глоссарий терминов

| Термин | Определение в контексте данного документа |
|--------|------------------------------------------|
| Network Effect (NE) | Ценность продукта для юзера растёт из-за активности / наличия / вклада других юзеров |
| Прямой NE | Больше юзеров → больше ценность для каждого (пример: мессенджер) |
| Непрямой NE | Одни юзеры создают ценность для других (пример: UGC, контекст split) |
| Core interaction | Минимальное действие, вокруг которого строится продукт и NE |
| Минимальный сетевой юнит | Наименьшая группа, в которой уже возникает ценность от NE |
| Кросс-кластерное распространение | Механизм, при котором юзер переносит продукт из одной группы в другую |
| Invitation-as-action | Приглашение, встроенное в core interaction (не отдельная реферальная кнопка) |
| Shared view | Совместная витрина финансовой информации группы поверх индивидуальных счетов |
| Contextual onboarding | Онбординг, при котором новый юзер попадает в готовый контекст, созданный другим юзером |
| Settlement rate | % долгов, закрытых за определённое время |
| Organic invite rate | % новых юзеров, привлечённых через core action, а не маркетинг |

### 24. Unit Economics + NE-driven экономика

**Формат:** модель юнит-экономики с учётом влияния NE на ключевые метрики

#### 24.1. Базовые допущения

| Параметр | Значение | Источник / обоснование |
|----------|---------|----------------------|
| Средний размер группы | 3.5 человека | Средневзвешенное: пары (2) + друзья (5) + семья (4) + соседи (3) |
| Групп на юзера (через 3 мес) | 2.2 | Целевое: мультигрупповое участие |
| Органическая конверсия приглашения | 35% | Benchmark Splitwise ~30%, наш target выше за счёт контекстного приглашения |
| Частота core action (split) | 4 раза / мес | 1 split/неделю в среднем по всем группам юзера |
| Средний чек split | 3 000₽ | Ресторан / коммуналка / поездка |
| Retention M1 | 65% | Benchmark финтех-приложений 40–50%, наш target выше за счёт NE |
| Retention M6 | 45% | С NE: группа держит юзера, даже если индивидуальная мотивация падает |

#### 24.2. Viral loop economics

 
1 юзер создаёт 1 группу (в среднем 3.5 участника)
    │
    ▼
Приглашает 2.5 новых юзеров (3.5 - 1 = 2.5, часть уже в системе)
    │
    ▼
Из 2.5 приглашённых → 35% конвертируются = 0.875 новых юзеров
    │
    ▼
Каждый новый юзер через 30 дней создаёт свою группу
    │
    ▼
Приглашает ещё 2.5 → конвертирует 0.875 → и так далее

K-factor (viral coefficient) = 2.5 × 0.35 = 0.875

⚠️ K < 1 → вирусный рост не самоподдерживающийся без внешнего вливания
Но: при мультигрупповом участии (2.2 группы/юзер):
    Effective K = 0.875 × 2.2 = 1.925 → K > 1 → самоподдерживающийся рост

⭐ Это ключевой инсайт: NE работает не через один viral loop, а через МУЛЬТИГРУППОВОЕ УЧАСТИЕ — один юзер = несколько циклов приглашения

#### 24.3. Как NE влияет на юнит-экономику

| Метрика | Без NE (обычный финтех) | С NE (наш продукт) | Почему разница |
|---------|------------------------|--------------------|----|
| **CAC** | $5–15 (paid acquisition) | $1–3 (organic invites = бесплатный канал привлечения) | Приглашение встроено в core action → organic acquisition |
| **Retention M1** | 40–50% | 60–70% | Уходить = бросить группу = социальное давление остаться |
| **Retention M6** | 20–30% | 40–50% | Накопленная история + ритуалы + множество групп |
| **LTV** | $15–30 | $40–80 | Дольше живёт + больше групп + больше действий |
| **LTV/CAC** | 2–3x | 15–25x | Кардинально другая экономика |
| **Payback period** | 3–6 мес | <1 мес | Низкий CAC + быстрая монетизация через freemium |

#### 24.4. Кривая ценности — как NE меняет экономику со временем

Ценность для юзера
▲
│                                          ╱ С NE: ценность растёт
│                                        ╱   (больше групп, больше
│                                      ╱     данных, больше людей)
│                                    ╱
│                                  ╱
│                    ╱────────────         Без NE: ценность стабильна
│                  ╱                      (продукт = инструмент)
│                ╱
│              ╱
│            ╱
│          ╱
│        ╱
│──────╱
└──────────────────────────────────────────► Время
     M1    M3     M6      M12     M24

**Вывод:** NE создаёт не просто лучшую юнит-экономику — он создаёт **улучшающуюся** юнит-экономику. Чем дольше юзер в продукте → тем ценнее продукт → тем ниже вероятность ухода → тем выше LTV. Это фундаментально другая бизнес-модель.

### 25. Сценарии развития — что если

**Формат:** три сценария с триггерами перехода

#### Сценарий A: «NE сработал» (вероятность: 30%)

Organic invite rate > 50%
Groups per user > 2.5
Retention M6 > 50%
K-factor effective > 1.5

Что происходит:
├── Самоподдерживающийся рост без маркетинга
├── Банки хотят интегрировать наш слой (B2B revenue)
├── Запуск marketplace (групповая покупательная способность)
├── Экспансия в новые гео
└── Раунд A/B на масштабирование

Стратегия: максимально быстро захватить сеть, пока окно открыто. Growth > profitability. Агрессивная экспансия в новые сегменты и гео.

#### Сценарий B: «NE слабый, но продукт полезен» (вероятность: 45%)

Organic invite rate 20–40%
Groups per user 1.2–1.8
Retention M6 30–40%
K-factor effective 0.7–1.0

Что происходит:
├── Продукт работает, но не растёт экспоненциально
├── NE есть, но недостаточно сильный для самоподдержания
├── Нужен маркетинг для поддержания роста
├── Юнит-экономика хорошая, но не выдающаяся
└── Pivot или усиление NE-механик

Стратегия:
├── Диагностика: КАКОЙ сегмент показывает лучший NE?
│   Сфокусироваться на нём
├── Усилить приглашение: что мешает конверсии?
├── Усилить мультигрупповое участие: почему юзеры
│   не создают вторую группу?
├── Рассмотреть B2B: встраивание в банковские
│   приложения как SDK
└── Рассмотреть pivot: от «финансовых групп»
    к «групповому финансовому ассистенту» (AI)

#### Сценарий C: «NE не сработал» (вероятность: 25%)

Organic invite rate < 20%
Groups per user < 1.2
Retention M6 < 25%
Нет разницы в retention между маленькими и большими группами

Что происходит:
├── NE не существует или слишком слабый
├── Продукт — утилитарный инструмент, а не сетевой продукт
├── Конкурентное преимущество отсутствует
├── Юнит-экономика как у обычного финтех-инструмента
└── Серьёзный pivot или закрытие

Стратегия:
├── Честная диагностика: ПОЧЕМУ не сработало?
│   ├── Боль недостаточно сильная?
│   │   (люди не делят деньги так часто)
│   ├── Культурный барьер непреодолим?
│   │   (о деньгах не говорят)
│   ├── Core interaction не тот?
│   │   (split — не лучший NE-trigger)
│   └── Архитектура не та?
│       (приглашение не встроено достаточно)
├── Варианты pivot:
│   ├── B2B SaaS: финансовые группы как фича
│   │   для банковских приложений
│   ├── Vertical focus: только пары (Honeydue 2.0)
│   │   или только микробизнес
│   └── Adjacent: от финансов к более широкой
│       групповой координации (не только деньги)
└── Kill criteria: если после pivot retention M3 < 30%
    → закрываем проект

### 29. Финальная мета-рефлексия

Формат: короткое эссе — «о чём этот документ на самом деле»

---

Этот документ — не план продукта. Это способ мышления о продукте.

Большинство финтех-продуктов проектируются от фичи: «давайте сделаем split», «давайте сделаем совместный счёт», «давайте сделаем бюджет для пар». Фича работает, но не создаёт moat. Конкурент копирует за 3 месяца. Банк добавляет кнопку за спринт.

Мы проектируем от сетевого эффекта: не «что будет делать юзер?», а «как действие одного юзера создаёт ценность для другого?»

Это принципиально другой вопрос. Он меняет:

- Архитектуру — группа, а не аккаунт как центральная сущность
- UX — приглашение как часть core action, а не отдельная кнопка
- Онбординг — контекст от старого юзера, а не пустое приложение
- Рост — органический через действия юзеров, а не купленный через маркетинг
- Retention — социальный (группа держит), а не индивидуальный (фичи держат)
- Moat — сетевой (невозможно воспроизвести без перетаскивания всей сети), а не продуктовый (скопировать за спринт)

Три вещи, которые я считаю ключевыми:

1. Минимальный сетевой юнит — это не миллион юзеров, а 3 друга за ужином. NE работает на уровне микрогруппы. Не нужна критическая масса в городе. Нужна критическая масса за одним столом. Это радикально снижает барьер запуска.

2. Мультигрупповое участие — это настоящий growth engine. Один юзер в 3 группах = 3 цикла приглашения = кросс-кластерное распространение. Именно это превращает K-factor < 1 в effective K-factor > 1.

3. Мы не знаем, сработает ли это. И это нормально. Мы знаем, как проверить. У нас есть конкретные гипотезы, конкретные эксперименты, конкретные pass/fail критерии. Через 12 недель мы будем знать ответ.

Самое ценное в этом документе — не идея. Идея очевидна (совместные финансы). Ценность — в архитектуре проверки: как от «мне кажется, это будет работать» дойти до «вот данные, которые подтверждают / опровергают NE».
