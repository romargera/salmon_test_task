# Ключевая рамка через весь документ

> Network effect отвечает на вопрос: растёт ли ценность продукта для пользователя из-за активности, наличия или вклада других пользователей?

Каждый раздел пропускается через этот фильтр. Если ценность не растёт от других пользователей — это не сетевой эффект, а что-то другое (масштаб, маркетинг, кросс-продажи, lock-in).

Два типа эффектов, которые мы различаем всюду:

- Прямой: больше пользователей → больше ценность для каждого (WhatsApp: «там мои родители, меня бесит, но я пользуюсь»)
- Непрямой: одни пользователи создают ценность для других (UGC-контент; «раздели счёт» — один юзер создаёт контекст, другие приходят за контекстом)

Два измерения пользователей: новые / старые — и нужно понимать, кто получает ценность от кого.

Рекомендуемый объем блока: 6 слайдов.

Общие указания для верстки блока:

- Это не теоретический блок, а блок выводов. На каждом слайде должна быть одна сильная мысль, а не пересказ исследований.
- На каждом слайде внизу должна быть отдельная контрастная плашка с формулировкой `Что это значит для нашего кейса`.
- Не перегружать таблицами. Лучше 3-5 коротких тезисов + один визуальный organizing principle.
- Ключевые слова, которые стоит визуально подсвечивать: `group`, `context`, `trust`, `invite`, `shared view`, `relationship-first`.
- Избегать академического тона. Формулировки должны звучать как продуктовые выводы для Salmon, а не как конспект литературы.
- Если блок нужно ужать до 5 слайдов, объединить слайды `B2.2` и `B2.3`.

## Слайд B2.1. Что делает продукт сетевым

Цель:
Поставить жесткую рамку: что считаем настоящим network effect, а что нет.

Заголовок:
`Network Effect (далее "NE") отвечает на вопрос: растёт ли ценность продукта для пользователя из-за активности, наличия или вклада других пользователей?`

Контент:

Проверка на NE:

- если убрать других людей, становится ли продукт принципиально слабее;
- если убрать маркетинг, продолжает ли рост поддерживаться самим использованием.

Три разных слоя:

- `Intent-driven NE`: пользователь приходит решить задачу, которая требует второго участника или становится сильно лучше при участии группы. Примеры: P2P-перевод, split bill, shared goal.
- `Context-driven NE`: один пользователь создает контекст, который сам по себе является ценностью для другого. Примеры: уже размеченный чек, запрос денег в контексте события, shared view, правила группы.
- `Behavioral-driven NE`: продукт становится полезнее, потому что нужные мне люди уже внутри, а поведение становится нормой. Это ближе к эффекту присутствия / стандарта, а не к отдельной core mechanic.

Что это значит для нашего кейса:
`Для Salmon стартовая ставка — intent-driven + context-driven внутри малой доверенной группы; behavioral layer должен появиться как следствие плотности и повторяемости, а не как обещание с первого дня.`

Визуал:
2 колонки `NE / Not NE`.

Указания для верстки:

- Композиция 2 колонки 55/45.
- Слева 4 карточки `НЕ network effect`: marketing, scale, lock-in, habit.
- Справа не список из "типов вперемешку", а 3 карточки: `Intent`, `Context`, `Behavior`.
- Внизу широкая плашка с takeaway.
- Не использовать длинные определения в 2-3 строки; каждая карточка должна читаться за 2 секунды.

## Слайд B2.2. Лучшие кейсы и общий паттерн

Цель:
Показать, что сильные NE-кейсы в финансах и рядом с финансами сходятся в core interaction, а не в отрасли.

Заголовок:
`Побеждают не платформы "на все случаи", а продукты с одной interaction, где другой человек необходим`

Контент:

- `Splitwise` — хороший context-driven кейс: один пользователь создает долг и структуру расчета, остальные приходят в уже существующий контекст.
- `M-Pesa` — сильный intent-driven кейс: задача перевода была настолько важной, что каждый успешный сценарий тянул в продукт следующего участника.
- `WhatsApp` — референс behavioral presence effect: продукт ценен, когда "нужные люди уже здесь", и именно это удерживает сеть.
- `Venmo / WeChat Pay` — гибрид: intent решает платежную задачу, context делает ее социально видимой, а behavioral layer появляется, когда это становится нормой внутри круга.
- Общий паттерн: сильные продукты не путают эти слои, а собирают их последовательно.

Что это значит для нашего кейса:
`Нам нужен wedge, где intent и context работают сразу, а behavioral pull появляется позже как следствие повторяющихся групповых действий.`

Визуал:
Сетка 2x2 из кейсов + нижняя строка `общий паттерн`.

Указания для верстки:

- На карточке каждого кейса только 3 строки: `какой слой`, `что создает ценность`, `что переносим`.
- Не превращать в большую сравнительную таблицу.
- Логотипы или пиктограммы можно использовать, но смысл должен читаться без них.
- Нижняя строка блока должна визуально объединять все карточки одним инсайтом.

## Слайд B2.3. Playbook: что реально переносится, а что нет

Цель:
Вытащить повторяющиеся правила запуска сетевого продукта и отделить архитектуру от тайминга и маркетинга.

Заголовок:
`Cold start выигрывает плотность микросети, а не ширина платформы`

Контент:

- Старт с одного узкого сценария, а не с "финансовой социальной сети".
- Ценность должна появляться уже в atomic network из 2-5 человек; плотность важнее масштаба.
- Invite должен быть частью core action: не бонус за друга, а невозможность получить полную ценность в одиночку.
- Cold start лучше запускать через существующую группу: пара, семья, поездка, соседи, чат друзей.
- Архитектура создает потенциал, но момент тоже важен: хорошие идеи без правильного тайминга и дистрибуции проваливаются.
- Маркетинг может купить первый трафик, но не может подменить network effect: если после отключения продвижения петля не живет сама, значит эффекта нет.

Что это значит для нашего кейса:
`Стартовая ставка должна быть на один повторяемый intent-driven сценарий, усиленный context-driven value, а не на широкий banking layer.`

Визуал:
Горизонтальный playbook из 5 шагов + маленький inset `Marketing vs NE`.

Указания для верстки:

- Верхние 2/3 слайда отдать под playbook-ленту из 5 шагов.
- Нижние 1/3 под компактное сравнение `купленный рост / self-reinforcing growth`.
- Хорошо работает стрелочная композиция слева направо.
- Не использовать больше 6 слов в названии каждого шага.

## Слайд B2.4. Где NE в финансах уже работает, а где до сих пор пусто

Цель:
Честно показать, что в финансах network effects уже есть, но в основном либо в инфраструктуре, либо в P2P, а не в everyday group money.

Заголовок:
`Финансы уже сетевые на уровне инфраструктуры, но почти не сетевые на уровне отношений между людьми`

Контент:

- Где NE уже работает: карточные сети, P2P-переводы, super-app payment loops, отдельные split/use-case механики.
- Где его почти нет: пары, семьи, соседи, travel-группы, бытовые shared expenses все еще живут в чатах, заметках и ручных переводах.
- Проблема не в отсутствии денег, а в отсутствии social layer поверх транзакций: группа, событие, справедливость, обещание, история.
- Банки проектируют мир как `user -> account`, а не как `user -> relationship -> group`.
- Ограничения реальны: KYC friction, privacy, страх социального давления, риск негативных NE при слишком больших группах.
- Практический лимит доверия для таких сценариев не бесконечен: максимальная сила эффекта обычно в малых группах, а не в массовой сети.

Что это значит для нашего кейса:
`Наш white space не "еще один кошелек", а relationship layer для trusted groups поверх существующих денег и счетов.`

Визуал:
Левая колонка `Где уже работает`, правая `Где пусто`, снизу строка `ограничения`.

Указания для верстки:

- Использовать асимметричную композицию: левая колонка компактнее, правая крупнее, чтобы визуально подчеркнуть white space.
- В нижней строке дать 3 ограничения через иконки: regulation, privacy, trust radius.
- Не делать slide слишком мрачным: справа должен чувствоваться opportunity, а не только gap.

## Слайд B2.5. White space: социальные деньги

Цель:
Сформулировать незанятое поле и показать, какие механики из других индустрий реально можно перенести в деньги.

Заголовок:
`White space: превратить денежное действие в социальный жест`

Контент:

- Социальные деньги это финансовое действие, которое одновременно меняет баланс и состояние отношений между людьми.
- Самые естественные офлайн-сценарии, которые до сих пор плохо оцифрованы: скинуться на подарок, разделить по справедливости, вести общий бытовой фонд, дать обещание или напоминание деньгами.
- Что переносим из соцсетей: group feed и реакции, но только в контексте своей группы, а не публичной ленты.
- Что переносим из маркетплейсов: matching между `кто заплатил`, `кто должен`, `кто подтверждает`.
- Что переносим из игр: streaks, progression, group challenges, unlocks для совместных целей.
- Что переносим из SaaS: shared view, роли, правила, единый workspace для группы.
- Главная адаптация под финансы: приватность, permission model и мягкая видимость важнее, чем максимальная социальность.

Что это значит для нашего кейса:
`Основа решения должна быть shared view + contextual split + recurring rituals; behavioral pull должен возникать из повторяемого использования, а не из искусственной social layer.`

Визуал:
Hub-and-spoke: в центре `Social Money`, вокруг 5 индустрий-переносов.

Указания для верстки:

- Центральный круг с одним сильным определением, вокруг 5 коротких spokes.
- На каждом spoke: `что берем` и `что адаптируем`.
- Лучше использовать не абстрактные названия индустрий, а конкретные механики: `feed`, `matching`, `streak`, `workspace`, `group context`.
- Внизу takeaway должен быть самым заметным элементом после заголовка.

## Слайд B2.6. Наша типология NE и архитектура успеха

Цель:
Собрать собственную классификацию без смешения осей и показать, какие продуктовые примитивы должны быть в MVP с первого дня.

Заголовок:
`Для Salmon нужен relationship-first coordination layer, где intent и context первичны, а behavior — следствие`

Контент:

- Мы различаем не один список "типов NE", а 3 слоя анализа: `intent-driven`, `context-driven`, `behavioral`.
- `Intent-driven` отвечает на вопрос: какая задача требует второго участника или группы.
- `Context-driven` отвечает на вопрос: какой контекст один участник создает для другого.
- `Behavioral` отвечает на вопрос: когда продукт начинает тянуть обратно, потому что нужные люди уже здесь и сценарий стал нормой.
- Для старта Salmon самые перспективные не platform и не data effects, а сочетание `intent-driven + context-driven`.
- Data, platform и standard effects можно показывать как later-stage amplifiers, а не как стартовую логику MVP.
- Базовая логика сущностей: `identity -> relationship -> group -> event -> shared view/rule -> reaction`.
- Relationship и Group должны быть first-class entities, иначе продукт снова скатится в обычный banking UX.
- Shared account не нужен на старте; shared view поверх существующих денег дает меньше трения и шире применим.
- Что не строить сразу: joint account, публичный feed, кредитный слой, broad marketplace, сложные reputation-системы.

Что это значит для нашего кейса:
`MVP должен сначала доказать силу task + context в малой группе; если это сработает, behavioral layer и более сильные moat-эффекты можно наращивать позже.`

Визуал:
Архитектурная цепочка сущностей + боковой блок `Build now / Later / Not now`.

Указания для верстки:

- Слева лучше показать 3-слойную схему `Intent / Context / Behavior`, а не сухой список типов.
- Справа сделать 3 коротких списка: `Now`, `Later`, `Do not start with`.
- Визуально выделить `relationship` и `group` как центральные узлы.
- Не использовать термин `shared wallet`, если он выглядит как полноценный банковский счет; лучше акцентировать именно `shared view`.
